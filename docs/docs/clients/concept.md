# Concept

A client in the context of Vivid is a program that uses features to control workflows or behavior.
If you're writing an application that reacts to a feature being enabled/disabled, or values of the metadata, you're a client.
Contrary to the web environment, a client can be anything; from a CLI to a backend application, from a browser page to a mobile app.

For interactions with Vivid, clients can use a specific client API.
A readonly API that allows them to fetch feature states.

## Concept

Each client will dispatch requests to a Vivid instance.
Every request from a client is done for a feature at an environment.
This api endpoint looks like this:

> /api/client/features/\{environment}/\{key}

:::note

The environment parameter can be either the `key`, `technical id` or `name` of the environment.
Vivid is able to resolve all of these as identifier for the environment.

If you can, use the `key`. This is more robust against changes in the environment.
You'll be able to find the key in the Vivid UI. Navigate to the environments page, open the environment you want to use, and copy the key.

![Environment Key](/img/env/select_env_key.png)

:::

:::warning

If the environment is not found, Vivid will return a 404 response.

:::

If successful, this endpoint will return the feature state of the feature identified by `key` in the environment `environment`.
You can now use the feature state to control your application.
But because this is a lot of repetitive work, Vivid provides a few SDKs to make this process easier:

## Streams

Clients can subscribe to feature streams of different technologies.
This allows them to adjust their behavior in real time when a feature changes.

Read more about streams in the [Streams Concept](streams).

## SDKs

Vivid provides a set of SDKs that can be used by your client to interact with Vivid.
These SDKs implement the Client API for different languages and frameworks.

You can read up on the available SDKs in the [SDKs Overview](sdks/overview).

## Security

The client api endpoints do not require OIDC authentication.
This is because the client SDKs are designed to be read-only endpoints.
However, we recognize that some companies may want to have more strict security requirements, even for Clients, and that this loosy-goosy open read api might not suite all needs.

Vivid provides a way to authenticate your client application.
Either you configure your Client to use an application name, or you can use a client token.
Both methods effectively achieve the same thing: a client application is explicitly identified against the Vivid instance.
They just differ in the way you set up clients.

:::note

Even if you are using a client token, or if you don't care that much about security in these endpoints, it is still recommended to provide an application name.
This will help you identify your client application in the Vivid UI.

Trust us, future you will thank you.

:::

### Client Name

An application can identify itself to Vivid by providing an application name.
This application name is transported in the `X-Application-Name` header, or in respective payload parameters.

Such an application name can be set to whatever you want.
In the spring client, you can set this property:

```properties
spring.vivid.application-name=my-client-name
```

With every request, Vivid will check if it has a matching application for the name you provided.
If it does, it will note down the action done by the client and make it available in the Vivid UI.
This makes it easier to identify if your client is connected to a given Vivid instance or when it last requested a feature state.

As an example, let's say you have a client named "test" and this client is connected to Vivid, using a feature named "Test".
If you navigate to the Vivid UI and open the "Test" feature, you'll see when the client "test" last requested the state of it:

![Client Uses Feature](/img/feature_used_by_client.png)

Similarly, you can navigate to the client details page and see details about when the client last requested the state of a feature or if it is considered online or offline:

![Client Details](/img/client_details_feature_usage.png)

This name makes it easier to identify the client and the feature it is using.

### Client Token

Client Token is a more "unique" way to identify a client.
It is a token that is generated by Vivid for each client.
This token is transported in the `X-Client-Token` header.

Whenever Vivid first sees a client, it will generate a client token for it.
This token is then used to identify the client in the Vivid UI.
You'll find it on the client details page.

![Client Details](/img/client_token.png)

### Additional Security - Restricting Access to the Client API

With the name and token, you can identify your client applications whenever they interact with Vivid.
But if you want, you can restrict access to unknown clients.
By default, Vivid allows any client that connects to the API and you can restrict that.

Navigate to the settings page and disable the setting "Allow Dynamic Client Registration".
By default, it will look like this:

![Client Settings](/img/allow_dynamic_client_registration_setting.png)

Once disabled, a new client will need to be registered manually in the Clients overview page before connecting to the API.
This is a good way to ensure that only authorized applications can connect to the API.

:::warning

Though more secure, this setting can have a negative impact on developer performance.

If you need to register every application that can connect to Vivid before it can go live, this setting is for you.
But keep in mind that this requires every developer to first create an application.
Pair this with reduced access rights, and developer satisfaction can suffer greatly because of this.

:::
